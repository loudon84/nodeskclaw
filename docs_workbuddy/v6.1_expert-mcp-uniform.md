# NoDeskClaw Expert MCP Gateway 统一调用规则调整 PRD

版本：v6.1
主题：专家 / 专家团队统一 MCP 调用规则
适用范围：nodeskclaw-backend、nodeskclaw-portal、copilot-desktop、Remote Hermes Expert Runtime
状态：实施版
基于版本：v6.0 Expert MCP Gateway
核心结论：专家与专家团队对 Desktop 暴露同一套 MCP 调用接口；区别只体现在 `annotations.kind` 与后台管理类型；默认由 upstream skill 自行完成团队内部智能体协作，nodeskclaw 只做网关、权限、日志、路由。

---

# 1. 背景

v6.0 已经实现 Expert MCP Gateway：

```text
GET  /api/v1/expert/health
POST /api/v1/expert/mcp
POST /api/v1/expert/mcp/{expert_slug}
```

并完成：

```text
1. Expert / ExpertSkill / ExpertTeam / ExpertTeamMember / ExpertInvocationLog 数据模型。
2. Expert Gateway 服务层。
3. 进程内复用 dispatch_agent_mcp 转发到底层 Hermes Agent。
4. /hermes/agents 页面专家状态、能力公开、调用开关、日志入口。
5. 专家调用日志。
6. expert_team_orchestrator 顺序调用成员专家并合并 Markdown。
```

v6.1 需要调整专家团队语义：

```text
旧语义：
  expert_team 默认由 nodeskclaw 后端编排成员专家。

新语义：
  expert_team 默认是一个“团队型专家入口”。
  它的 skill 可以在 Hermes Agent 内部自行调用多个智能体。
  nodeskclaw 默认不编排成员专家，只做网关、权限、日志、路由。
```

---

# 2. v6.1 核心目标

## 2.1 产品目标

统一 Desktop 调用体验：

```text
1. Desktop 调用专家和专家团队使用同一个接口。
2. Desktop 不需要知道 slug 背后是 expert 还是 expert_team。
3. Desktop 只根据 annotations.kind 做 UI 展示区分。
4. 专家团队默认由 upstream skill 内部完成多智能体协作。
5. nodeskclaw 不默认执行 expert_team_orchestrator。
6. 已实现的 expert_team_orchestrator 保留为可选高级模式。
```

## 2.2 技术目标

统一调用入口：

```text
POST /api/v1/expert/mcp/{slug}
```

其中：

```text
slug 可以是：
  expert_slug
  team_slug
```

统一处理逻辑：

```text
Desktop
  ↓
POST /api/v1/expert/mcp/{slug}
  ↓
Expert MCP Gateway
  ↓
CatalogService.resolve(slug)
  ↓
根据 catalog.kind 判断 expert / expert_team
  ↓
默认 upstream_skill 模式：转发到 /api/v1/hermes/mcp/{agent_profile}
  ↓
Hermes Agent skill 执行
```

---

# 3. 不做范围

v6.1 不做：

```text
1. 不引入 HermesTask。
2. 不引入队列 / Worker。
3. 不默认启用 nodeskclaw 团队编排。
4. 不要求 Desktop 消费 team.member.* timeline。
5. 不要求 Desktop 区分 callExpert 与 callExpertTeam。
6. 不要求 Desktop 直接调用成员专家。
7. 不要求 Desktop 传 runtime、agent_alias、route_config。
8. 不改变底层 Hermes Agent 的 skill 执行协议。
```

---

# 4. 最终规则

```text
1. 专家与专家团队的 Desktop 调用接口一致。
2. 都走 POST /api/v1/expert/mcp/{slug}。
3. 区别只体现在 annotations.kind 和后台管理类型。
4. 专家团队不一定需要 nodeskclaw 编排。
5. 如果团队 skill 自己能调用多个智能体，nodeskclaw 只做网关、权限、日志和路由。
6. 当前 Cursor 实现中有 expert_team_orchestrator，需要改为可选模式或默认不启用。
```

固定为：

```text
Desktop 统一调用：
  POST /api/v1/expert/mcp/{slug}

slug 类型：
  expert
  expert_team

调用差异：
  Desktop 无差异
  nodeskclaw 只根据 kind 做展示和权限策略
  具体协作逻辑由 upstream skill 自己完成
```

# 5. 概念调整

## 5.1 Catalog Item

新增统一抽象：

```text
Catalog Item = Expert 或 ExpertTeam 的统一产品目录项
```

字段：

```text
slug
kind: expert | expert_team
display_name
description
category
tags
published
enabled
hermes_agent_id
orchestration_mode
```

## 5.2 slug

v6.0 中接口变量名为：

```text
expert_slug
```

v6.1 改为：

```text
slug
catalog_slug
```

原因：

```text
1. slug 可能指向 expert。
2. slug 也可能指向 expert_team。
3. Desktop 不应关心具体后台表类型。
```

保留兼容：

```text
旧路径：
  POST /api/v1/expert/mcp/{expert_slug}

新语义：
  POST /api/v1/expert/mcp/{slug}
```

实际 URL 不变，只调整代码变量、文档与服务层语义。

## 5.3 annotations.kind

统一使用：

```text
expert
expert_skill
expert_team
expert_team_skill
```

Desktop 根据 `annotations.kind` 决定 UI 展示类型，不决定调用路径。

## 5.4 orchestration_mode

专家团队新增或调整字段：

```text
orchestration_mode:
  upstream_skill
  gateway_sequential
  gateway_parallel_reserved
```

默认：

```text
upstream_skill
```

含义：

```text
upstream_skill:
  nodeskclaw 不编排团队，直接调用 upstream skill。
  upstream skill 自行完成多智能体协作。

gateway_sequential:
  使用当前已实现的 expert_team_orchestrator。
  nodeskclaw 顺序调用成员专家并合并 Markdown。
  这是可选高级模式，不是默认模式。

gateway_parallel_reserved:
  预留，不在 v6.1 实现。
```

# 8. 后端调用流程

## 8.1 统一 resolve 流程

```text
POST /api/v1/expert/mcp/{slug}
  ↓
CatalogResolver.resolve(slug)
  ↓
返回 CatalogItem:
  kind = expert | expert_team
  slug
  hermes_agent_id
  orchestration_mode
  skill table source
```

查询顺序建议：

```text
1. 查 experts.expert_slug = slug。
2. 找不到时查 expert_teams.team_slug = slug。
3. 若仍找不到，返回 EXPERT_CATALOG_NOT_FOUND。
```

也可以后续收敛为统一表：

```text
expert_catalog_items
```

但 v6.1 不强制大改数据模型。

## 8.2 默认 upstream_skill 调用

```text
CatalogItem.kind = expert
  ↓
查 expert_skills
  ↓
找到 skill_name -> upstream_tool_name
  ↓
dispatch_agent_mcp(profile_name, tools/call)

CatalogItem.kind = expert_team
orchestration_mode = upstream_skill
  ↓
查 expert_team_skills 或复用 expert_skills-like skill 表
  ↓
找到 skill_name -> upstream_tool_name
  ↓
dispatch_agent_mcp(profile_name, tools/call)
```

说明：

```text
1. expert_team 默认不调用 expert_team_orchestrator。
2. team skill 的内部多智能体协作由 upstream skill 自己完成。
3. nodeskclaw 不拆解团队成员执行过程。
```

## 8.3 可选 gateway_sequential 调用

```text
CatalogItem.kind = expert_team
orchestration_mode = gateway_sequential
  ↓
调用 expert_team_orchestrator
  ↓
顺序调用成员 expert skill
  ↓
合并 Markdown
  ↓
返回结果
```

规则：

```text
1. gateway_sequential 不是默认模式。
2. 只有后台明确设置 orchestration_mode=gateway_sequential 时才启用。
3. 未配置成员专家时禁止启用 gateway_sequential。
4. gateway_sequential 保留现有父子 invocation log。
```

# 9. 数据模型调整

## 9.1 保持 v6.0 表结构，新增兼容字段

v6.1 尽量不破坏 v6.0 已完成的数据模型。

现有表：

```text
experts
expert_skills
expert_teams
expert_team_members
expert_invocation_logs
```

建议新增或确认字段：

### expert_teams

```sql
ALTER TABLE expert_teams
  ADD COLUMN IF NOT EXISTS hermes_agent_id UUID NULL;

ALTER TABLE expert_teams
  ADD COLUMN IF NOT EXISTS orchestration_mode VARCHAR(64) DEFAULT 'upstream_skill';

ALTER TABLE expert_teams
  ADD COLUMN IF NOT EXISTS upstream_tool_name VARCHAR(255) NULL;

ALTER TABLE expert_teams
  ADD COLUMN IF NOT EXISTS input_schema JSONB DEFAULT '{}';

ALTER TABLE expert_teams
  ADD COLUMN IF NOT EXISTS output_formats JSONB DEFAULT '["markdown"]';
```

说明：

```text
hermes_agent_id:
  team skill 所在 Hermes Agent。

orchestration_mode:
  upstream_skill | gateway_sequential | gateway_parallel_reserved。

upstream_tool_name:
  默认 upstream_skill 模式下，团队入口 skill 对应的真实 MCP tool name。

input_schema:
  team skill 的输入 schema。

output_formats:
  输出格式。
```

如果你不希望把 skill 字段塞在 expert_teams 表，也可以新增：

```text
expert_team_skills
```

推荐更规范做法如下。

## 9.2 推荐新增 expert_team_skills 表

```sql
CREATE TABLE IF NOT EXISTS expert_team_skills (
  id UUID PRIMARY KEY,
  org_id UUID NOT NULL,
  expert_team_id UUID NOT NULL,

  skill_name VARCHAR(128) NOT NULL,
  upstream_tool_name VARCHAR(255) NOT NULL,

  display_name VARCHAR(128),
  description TEXT,
  input_schema JSONB DEFAULT '{}',

  public BOOLEAN DEFAULT false,
  call_enabled BOOLEAN DEFAULT false,

  risk_level VARCHAR(32) DEFAULT 'low',
  approval_mode VARCHAR(32) DEFAULT 'server',
  output_formats JSONB DEFAULT '["markdown"]',

  stale BOOLEAN DEFAULT false,
  last_synced_at TIMESTAMP,

  sort_order INT DEFAULT 100,

  created_by UUID,
  updated_by UUID,
  created_at TIMESTAMP NOT NULL,
  updated_at TIMESTAMP NOT NULL,
  deleted_at TIMESTAMP NULL
);
```

唯一索引：

```sql
CREATE UNIQUE INDEX IF NOT EXISTS uq_expert_team_skills_slug
ON expert_team_skills(org_id, expert_team_id, skill_name)
WHERE deleted_at IS NULL;

CREATE UNIQUE INDEX IF NOT EXISTS uq_expert_team_skills_upstream
ON expert_team_skills(org_id, expert_team_id, upstream_tool_name)
WHERE deleted_at IS NULL;
```

# 10. 服务层调整

## 10.1 新增 CatalogResolver

文件建议：

```text
app/services/expert_gateway/catalog_resolver.py
```

职责：

```text
1. resolve(slug)
2. find expert by expert_slug
3. find expert_team by team_slug
4. 返回统一 CatalogItem DTO
5. 屏蔽底层表差异
```

DTO：

```python
class CatalogItem:
    id: UUID
    org_id: UUID
    kind: Literal["expert", "expert_team"]
    slug: str
    display_name: str
    description: str | None
    hermes_agent_id: UUID | None
    published: bool
    enabled: bool
    orchestration_mode: str
```

## 10.2 调整 expert_mcp_gateway_service

当前 v6.0：

```text
dispatch_root
dispatch_expert
```

v6.1 调整为：

```text
dispatch_root
dispatch_catalog_item
```

接口函数命名：

```python
dispatch_catalog_item(slug, jsonrpc_body, user, org)
```

处理：

```text
initialize:
  返回 MCP initialize result。

tools/list:
  根据 CatalogItem.kind 返回 expert_skill 或 expert_team_skill。

tools/call:
  统一校验权限、public、call_enabled、route guard、日志。
  然后按 orchestration_mode 分派。
```

## 10.3 调整 expert_mcp_proxy_service

保持 v6.0 已有逻辑：

```text
构造上游 body:
{
  "method": "tools/call",
  "params": {
    "name": upstream_tool_name,
    "arguments": arguments
  }
}

进程内调用:
dispatch_agent_mcp(db, org_id, user_id, profile_name, body)
```

v6.1 补充：

```text
1. profile_name 从 CatalogItem.hermes_agent_id 找到 HermesAgentInstance.profile_name。
2. 不允许 Desktop 传 agent_alias 或 runtime。
3. expert_team upstream_skill 模式也走同一个 proxy。
```

## 10.4 调整 expert_team_orchestrator

当前实现：

```text
默认参与 expert_team tools/call。
```

v6.1 修改：

```text
只在 orchestration_mode = gateway_sequential 时参与。
```

要求：

```text
1. 不删除代码。
2. 不作为默认调用路径。
```
