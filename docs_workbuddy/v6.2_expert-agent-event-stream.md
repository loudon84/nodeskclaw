# NoDeskClaw PRD v6.2

## Expert MCP 实时 Agent Event Stream 任务窗口改造方案

版本：v6.2
项目：nodeskclaw
关联前端：copilot-desktop
目标模块：Expert MCP Gateway、HermesTask、Hermes Agent Runtime、Task SSE、Artifacts、Expert Team
实施工具：Cursor
日期：2026-06-28

---

# 1. 背景

NoDeskClaw 当前已经具备 Expert Catalog、Expert MCP Gateway、Hermes Agent Runtime 状态管理、HermesTask 任务系统、Task SSE 事件流、Artifacts pull-only 成果中心等基础能力。

当前 Expert MCP Gateway 的主要调用链仍偏同步：

```text
copilot-desktop / WebUI
  -> POST /api/v1/expert/mcp/{slug}
  -> tools/call
  -> ExpertMcpGatewayService
  -> ExpertMcpProxyService.call_upstream_tool()
  -> per-agent Hermes MCP Gateway
  -> Hermes API_SERVER /v1/chat/completions
  -> 等完整结果返回
```

该模式适合一次性返回报告，但不适合实现类似腾讯 WorkBuddy 的“任务窗口”体验。WorkBuddy 的核心体验是：

```text
任务启动后立即进入任务窗口
  -> 多专家状态实时显示
  -> Agent 过程实时 streaming
  -> 阶段事件持续追加
  -> 右侧成果文件实时出现
  -> 任务结束后统一沉淀报告和 artifacts
```

因此 v6.2 的核心目标是：

```text
将 Expert MCP tools/call 从“同步 RPC 拿结果”
切换为“创建 HermesTask + 订阅 Agent Event Stream + 产物 pull-only”
```

---

# 2. 产品目标

## 2.1 总目标

在 NoDeskClaw 中实现 Expert MCP 实时任务调用能力，使 copilot-desktop 可以像 WorkBuddy 一样，在任务窗口中实时展示专家 / 专家团队的 Agent 执行过程。

## 2.2 用户目标

用户在 copilot-desktop 中选择一个专家或专家团队，输入任务后：

```text
1. 立即看到任务已启动。
2. 立即进入任务窗口。
3. 看到每个专家成员的状态变化。
4. 看到 Agent 实时输出和阶段进度。
5. 看到工具调用、分析步骤、产物生成事件。
6. 任务完成后看到最终报告和成果文件。
7. 可取消、重试、查看历史、下载成果。
```

## 2.3 平台目标

NoDeskClaw 后端形成统一的 Expert Run 模型：

```text
Expert MCP tools/call
  -> ExpertRun
  -> HermesTask
  -> HermesTaskEvent
  -> Task SSE
  -> Artifacts
```

所有专家调用、专家团队调用、Desktop 任务窗口展示、后台调用日志，都围绕 `HermesTask` 和 `HermesTaskEvent` 统一。

# 4. 核心设计原则

## 4.1 Expert MCP 只负责启动任务

v6.2 后，`tools/call` 默认不再等待完整报告。

```text
旧模式：
tools/call -> 完整结果

新模式：
tools/call -> task_id + event_url + artifact_url
```

## 4.2 路由权必须留在服务端

客户端不得通过参数覆盖 Hermes Agent、profile、workspace、gateway_url、tool route 等路由信息。

继续禁止以下参数：

```text
profile
agent_alias
agent_id
workspace
workspace_id
gateway_url
route
routing
_execution
_route
route_config
```

这些信息只能由 Expert Catalog、ExpertSkill、ExpertTeam、HermesAgentInstance、Runtime 状态和服务端策略决定。

## 4.3 统一使用 HermesTask 承载 Expert Run

Expert Run 不新增一套任务表。复用现有 `HermesTask`：

```text
HermesTask
  -> status
  -> event_url
  -> artifact_url
  -> routing_metadata
  -> client_context
  -> output_policy
  -> server_artifacts
```

## 4.4 统一使用 Task SSE 输出事件

copilot-desktop 只订阅：

```text
GET /api/v1/hermes/tasks/{task_id}/events
```

而不是直接订阅每个 Hermes Agent 的原生 API_SERVER。

## 4.5 产物继续 pull-only

Expert 执行产生的报告、Markdown、JSON、PDF、DOCX 等成果，统一进入 NoDeskClaw server_artifacts，由前端预览、下载、导入。

# 7. 后端功能需求

# 7.1 Expert MCP tools/call 异步化

## 7.1.1 当前行为

当前 `_call_expert_skill()` 会同步调用上游 Hermes API_SERVER，等待完整结果后返回。

## 7.1.2 目标行为

默认改为：

```text
tools/call
  -> validate expert / team / skill
  -> validate permission
  -> validate runtime ready
  -> create HermesTask
  -> create ExpertInvocationLog
  -> create event token
  -> return accepted result
```

## 7.1.3 兼容策略

新增 request 参数或 header：

```text
X-NoDeskClaw-Expert-Run-Mode: event_stream | sync_legacy
```

默认：

```text
event_stream
```

兼容：

```text
sync_legacy
```

`sync_legacy` 仅用于 Postman、旧客户端、调试，不作为 Desktop 默认路径。

## 7.1.4 返回格式

MCP `tools/call` 成功后返回：


# 7.3 HermesTaskWorker 增加 Expert Event Stream 分支

## 7.3.1 当前分支

当前 Worker 已经识别：

```text
route_type == "hermes_api_server"
```

并进入 `_execute_api_server_task()`。

## 7.3.2 新增分支

新增：

```python
if route_snapshot.get("route_type") == "expert_agent_event_stream":
    await self._execute_expert_agent_event_stream_task(...)
    return
```

## 7.3.3 新增方法

文件：

```text
nodeskclaw-backend/app/services/hermes_skill/hermes_task_worker.py
```

新增：

```python
async def _execute_expert_agent_event_stream_task(
    self,
    db: AsyncSession,
    task: HermesTask,
    route_snapshot: dict,
    task_service: TaskService,
    event_service: TaskEventService,
    audit_logger: SkillAuditLogger,
) -> None:
    ...
```

## 7.3.4 执行流程

```text
1. 解析 route_snapshot。
2. 校验 Hermes Agent 绑定仍有效。
3. 写入 HERMES_RUN_STARTED event。
4. 使用 HermesAgentAdapter.submit_run(task, task.arguments)。
5. 保存 task.hermes_run_id。
6. 写入 HERMES_RUN_CREATED event。
7. 使用 HermesAgentAdapter.read_run_events(task) 订阅远端事件。
8. 将远端事件转换成 HermesTaskEvent。
9. 写入 event_service。
10. 发布 EventBus，驱动 SSE。
11. 监听 completed / failed / cancelled / timeout。
12. completed 后执行 artifact scan / materialize。
13. 写入 task.completed。
14. publish_completed_with_result。
```

## 7.3.5 route_snapshot 示例

```json
{
  "route_type": "expert_agent_event_stream",
  "catalog_kind": "expert",
  "catalog_slug": "call-prep",
  "orchestration_mode": "agent_event_stream",
  "hermes_agent_instance_id": "agent-id",
  "agent_profile": "common-researcher",
  "runtime_skill_id": "customer-profiling",
  "upstream_tool_name": "hermes_common_researcher__customer-profiling",
  "hermes_instance_name": "hermes-common-researcher",
  "timeout_seconds": 1800
}
```

# 7.8 取消任务

现有任务取消 API：

```text
POST /api/v1/hermes/tasks/{task_id}/cancel
```

v6.2 要求：

```text
1. 如果 task.hermes_run_id 存在，调用 HermesAgentAdapter.cancel_run(task)。
2. 写入 task.cancel_requested。
3. 最终写入 task.cancelled。
4. SSE 输出 task.failed 或 task.cancelled。
```

Desktop 显示：

```text
用户已取消
```

# 13. 安全要求

```text
1. Desktop 不允许传 route override 参数。
2. eventToken 只绑定 task_id。
3. eventToken 有过期时间。
4. eventToken 不可访问 artifacts 下载接口，artifact 仍需正常鉴权或签名 URL。
5. Expert MCP tools/call 必须校验 expert_skill:invoke。
6. tools/list 必须校验 expert:view / expert_skill:view。
7. 非 admin/operator 用户只能查看自己有权限的任务。
8. invocation log 中隐藏 token / password / api_key。
```

---

# 14. 性能要求

```text
1. tools/call 启动任务响应时间 < 1 秒。
2. SSE 首个 task.started 事件 < 2 秒。
3. heartbeat 默认沿用现有配置。
4. 单个用户同时最多运行任务数遵循 HermesQueuePolicy。
5. 单 Agent 并发遵循 HermesAgentRuntimeState.max_concurrent_tasks。
6. Worker 不应因单个长任务阻塞整个队列。
```

```text
  -> 选择专家 / 专家团队
  -> POST /api/v1/expert/mcp/{slug} tools/call
```
