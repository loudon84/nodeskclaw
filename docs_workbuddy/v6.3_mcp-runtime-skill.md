# NoDeskClaw 修复方案 PRD v6.3

版本：v6.3
主题：Desktop 统一通过 NoDeskClaw MCP `tools/call` 调用 Runtime Skill 固定路由，并启用 SSE Timeline
适用范围：`nodeskclaw-backend`、`mcp_skill_gateway`、`hermes_skill`、`HermesTaskWorker`、Desktop v7.5 SSE Timeline
状态：修复型 PRD / Implementation Guide

---

## 1. 背景与问题

Desktop v7.5 已启动 SSE Timeline，但当前存在两类执行路径混用问题：

```text
路径 A：Desktop / Expert Gateway
  → expert_agent_event_stream
  → Hermes Agent /v1/runs
  → /v1/runs/{run_id}/events
  → 依赖 Hermes Run 事件终态

路径 B：Hermes MCP Runtime Skill
  → NoDeskClaw MCP tools/call
  → hermes_api_server 固定路由
  → Hermes Agent /v1/chat/completions
  → NoDeskClaw 任务事件 + 结果 + 中心产物
```

当前目标是统一为：

```text
Desktop v7.5
  → NoDeskClaw MCP tools/call
  → Runtime Skill 固定路由 hermes_api_server
  → NoDeskClaw HermesTask
  → NoDeskClaw SSE Timeline
  → Result / Artifacts / KB 状态
```

源码上，NoDeskClaw 的 MCP Gateway 已经以 `initialize`、`tools/list`、`tools/call` 作为统一入口；`tools/call` 会进入 `_handle_tools_call()`，再调用 `McpToolMapper.call_tool()` 创建任务。

任务执行层已经存在分流逻辑：Worker 根据 `route_snapshot.route_type` 判断是否走 `hermes_api_server`；如果是，则执行 `_execute_api_server_task()`，否则进入 `agent_run_stream` 路径。

## 2. 核心目标

### 2.1 产品目标

让 Desktop v7.5 的专家调用具备以下稳定体验：

```text
1. 用户在 Desktop 选择专家 / Skill。
2. Desktop 只调用 NoDeskClaw MCP tools/call。
3. NoDeskClaw 后端固定选择 Runtime Skill 绑定的 Hermes 实例。
4. Desktop 不传 _routing / _execution / route_config。
5. 任务创建后立即返回 task_id、event_stream、result_url、artifact_url。
6. Desktop 订阅 NoDeskClaw /hermes/tasks/{task_id}/events。
7. 运行中即可看到 Timeline，不需要等 timeout 才能查看日志。
8. 完成后展示 result、server_artifacts、KB 状态。
```

### 2.2 技术目标

```text
1. Runtime Skill 统一使用 hermes_api_server 固定路由。
2. SSE Timeline 由 NoDeskClaw HermesTask 事件系统提供。
3. Desktop 不再直接依赖 Hermes Agent /v1/runs 事件流。
4. MCP tools/list 返回足够的 Runtime Skill 元数据，让 Desktop 能识别可调用专家。
5. MCP tools/call 对 Runtime Skill 始终返回 async_event 结构。
6. Task Detail / Timeline / Result API 在 running 状态下可安全读取。
7. 任务超时、事件断流、结果未就绪都返回结构化状态，不让前端卡死。
```

---

## 3. 非目标

v6.3 不做以下内容：

```text
1. 不改 Desktop 的产品交互布局。
2. 不恢复 Desktop 旧 Expert Gateway 强制路由。
3. 不允许 Desktop 传 _routing / _execution / route_config。
4. 不把 Hermes Agent /v1/runs 作为 Runtime Skill 默认执行方式。
5. 不要求 /v1/chat/completions 提供 token 级流式输出。
6. 不把 server_artifacts 伪装成本地文件。
7. 不在 Desktop 编排专家团队成员。
```

## 4. 当前源码依据

### 4.1 MCP Gateway 已有统一入口

`/hermes/mcp` 路由由 `mcp_skill_gateway_router` 提供，后端 Router 已把 MCP Skill Gateway 与 Hermes Skill Router 注册到 `/api/v1` 下。

MCP Gateway 中 `tools/call` 会进入 `_handle_tools_call()`，再进入 `McpToolMapper.call_tool()`。

### 4.2 Runtime Skill 已有固定路由策略

当 `skill.source_type == "hermes_api_server"` 时，当前代码已经禁止调用方覆盖路由。如果 arguments 包含 `_routing`、`_execution`、`route_config`，会抛出“组织级 MCP 不允许覆盖 Hermes 实例路由”。

固定路由会调用 `resolve_runtime_skill_fixed_route()`，并要求 installation 的 `routing_metadata` 必须包含：

```text
route_type = hermes_api_server
force_instance = true
hermes_agent_instance_id
agent_profile
runtime_skill_id
```

源码已有校验。

### 4.3 NoDeskClaw 已有 SSE Timeline 基础

`/hermes/tasks/{task_id}/events` 已支持 Bearer Token 或 SSE token 访问，支持 `Last-Event-ID` 恢复，支持 heartbeat，并在终态时输出 timeline snapshot。

任务完成事件还会在需要时附加 result 数据。

### 4.4 Task Result / Event Token 已有 API

`POST /hermes/tasks/{task_id}/events-token` 可创建 SSE token，`GET /hermes/tasks/{task_id}/result` 可读取任务结果。

## 6. 核心流程

### 6.1 Desktop 发现 Runtime Skill

```text
Desktop
  → MCP initialize
  → MCP tools/list
  → 过滤 sourceType = hermes_api_server
  → 过滤 routeType = hermes_api_server
  → 渲染专家 / Skill 卡片
```

`tools/list` 当前已从 `McpToolMapper.list_tools()` 查询安装状态、授权状态、agent/profile/workspace 信息。

v6.3 要求增强返回元数据：

```json
{
  "name": "hermes_xieyi__customer-profiling",
  "title": "客户画像分析",
  "sourceType": "hermes_api_server",
  "routeType": "hermes_api_server",
  "serverManagedRoute": true,
  "runtimeSkillId": "customer-profiling",
  "agentId": "...",
  "profileId": "...",
  "workspaceId": "...",
  "executionModes": ["async_event"],
  "sseTimelineEnabled": true,
  "artifactMode": "pull_only",
  "requiresRouteOverride": false,
  "forbiddenArgumentKeys": ["_routing", "_execution", "route_config"]
}
```

### 6.2 Desktop 调用 Runtime Skill

请求必须保持简单：

```json
{
  "jsonrpc": "2.0",
  "id": "call_001",
  "method": "tools/call",
  "params": {
    "name": "hermes_xieyi__customer-profiling",
    "arguments": {
      "prompt": "分析研华科技的客户画像、采购动机和潜在销售机会。",
      "context": {
        "source": "copilot-desktop",
        "device_id": "desktop-001",
        "conversation_id": "local-session-001",
        "workspace_id": "default"
      }
    }
  }
}
```

禁止：

```json
{
  "_routing": {},
  "_execution": {},
  "route_config": {}
}
```

NoDeskClaw 接收到 `tools/call` 后：

```text
1. 校验 tool_name。
2. strip _wait 控制参数。
3. 禁止 Runtime Skill route override。
4. resolve_runtime_skill_fixed_route。
5. 创建 HermesTask。
6. 写入 routing_metadata.route_snapshot。
7. 返回 async_event 响应。
```

### 6.3 NoDeskClaw 返回 async_event

当前源码中 async_event 响应已经包含 `event_stream`、`event_token_url`、`artifact_url`、`result_url`、`wait_strategy` 等字段。

v6.3 要求 Runtime Skill 对 Desktop 固定返回：

```json
{
  "tool_name": "hermes_xieyi__customer-profiling",
  "task_id": "...",
  "task_no": "TASK-2be7-xxxx",
  "status": "running",
  "execution_mode": "async_event",
  "event_stream": "http://.../api/v1/hermes/tasks/{task_id}/events?token=...",
  "event_token_url": "/api/v1/hermes/tasks/{task_id}/events-token",
  "result_url": "/api/v1/hermes/tasks/{task_id}/result",
  "artifact_url": "/api/v1/hermes/tasks/{task_id}/artifacts",
  "artifact_mode": "pull_only",
  "server_artifacts": [],
  "wait_strategy": {
    "type": "sse",
    "fallback": "poll",
    "poll_url": "/api/v1/hermes/tasks/{task_id}",
    "result_url": "/api/v1/hermes/tasks/{task_id}/result"
  },
  "committed": true
}
```

## 7. 功能需求

### F1. Runtime Skill 固定路由强约束

现状已有禁止 route override 的代码。v6.3 需要把它产品化、可诊断化。

要求：

```text
1. hermes_api_server 类型 Skill 必须走固定 installation。
2. 多个 installation 时必须有唯一 default。
3. route_config 必须含 route_type=hermes_api_server、force_instance=true、hermes_agent_instance_id、agent_profile、runtime_skill_id。
4. Desktop 传 _routing / _execution / route_config 时直接返回结构化错误。
5. 错误 data 中返回 override_keys、tool_name、expected_mode、suggested_arguments。
```

建议错误响应：

```json
{
  "error": {
    "code": -32602,
    "message": "组织级 MCP 不允许覆盖 Hermes 实例路由",
    "data": {
      "errorCode": "errors.skill.route_override_not_allowed",
      "tool_name": "hermes_xieyi__customer-profiling",
      "override_keys": ["_routing"],
      "expected_mode": "server_managed_fixed_route",
      "suggested_arguments": {
        "prompt": "string",
        "context": "object"
      }
    }
  }
}
```

### F2. tools/list 增强 Runtime Skill 元数据

当前 `_skill_to_tool_dict()` 已返回 `name/title/description/inputSchema/agentId/profileId/workspaceId/approvalMode/authorized` 等字段。

v6.3 增加：

```text
sourceType
routeType
serverManagedRoute
runtimeSkillId
runtimeInstanceId
runtimeInstanceName
runtimeProfile
executionModes
defaultExecutionMode
sseTimelineEnabled
eventStreamProvider
artifactMode
resultMode
routeOverrideAllowed
forbiddenArgumentKeys
routeHealth
```

用于 Desktop 判断：

```text
1. 是否显示为远端专家。
2. 是否可以调用。
3. 是否支持 SSE Timeline。
4. 是否需要结果页 fallback polling。
5. 是否禁止传路由字段。
```

### F3. tools/call 默认 async_event

`resolve_mcp_execution_mode()` 当前已支持 default async_event，且 `mcp_client_token` 会返回 `ASYNC_EVENT_MODE`。

v6.3 要求：

```text
1. Desktop 使用 mcp_client_token 时，Runtime Skill 默认 async_event。
2. user_jwt 调用 Runtime Skill 可继续 queued，但 Desktop 专用 token 必须 async_event。
3. _wait=true 仅保留给调试，不作为 Desktop 正式调用方式。
4. _wait=false 不得影响任务创建后的 SSE Timeline 可用性。
```

### F4. Task Snapshot API：运行中也能打开明细

当前已有：

```text
GET /hermes/tasks/{task_id}
GET /hermes/tasks/{task_id}/timeline
GET /hermes/tasks/{task_id}/events
GET /hermes/tasks/{task_id}/result
GET /hermes/tasks/{task_id}/artifacts
```

v6.3 新增聚合接口，避免 Desktop 明细弹窗在 running 状态下因 result 未完成而卡住：

```text
GET /api/v1/hermes/tasks/{task_id}/snapshot
```

响应：

```json
{
  "task": {},
  "status": "running",
  "timeline": [],
  "result": {
    "ready": false,
    "summary": null,
    "content": null
  },
  "artifacts": {
    "ready": false,
    "items": [],
    "server_artifacts": []
  },
  "links": {
    "event_stream": "...",
    "result_url": "...",
    "artifact_url": "..."
  }
}
```

规则：

```text
1. running / queued / accepted 状态返回 200。
2. result 未就绪时返回 ready=false，不抛 404。
3. artifacts 未生成时返回空数组，不抛异常。
4. failed / timeout 时返回 error_code、error_message、last_events。
5. completed 时返回 result.ready=true 和 artifacts.ready=true。
```

### F5. SSE Timeline 事件标准化

当前事件流支持已有事件、heartbeat、终态 timeline snapshot。

v6.3 要求标准事件类型：

```text
task.created
task.queued
task.accepted
task.running
hermes.run.started
hermes.run.completed
hermes.run.failed
artifact.ready
task.completed
task.failed
task.timeout
task.timeline
```

Runtime Skill 走 `/v1/chat/completions` 时，没有 token 级 delta，也必须输出基础 Timeline：

```text
任务创建 → 入队 → Worker 接收 → Hermes 实例绑定 → Runtime Skill 开始 → Runtime Skill 完成 → 产物生成 → 任务完成
```

### F6. Result API 非终态兼容

`GET /tasks/{task_id}/result` 当前适合完成后读取。v6.3 要求增强为非终态安全：

```json
{
  "ready": false,
  "status": "running",
  "task_id": "...",
  "task_no": "...",
  "message": "任务仍在执行中",
  "result_summary": null,
  "content": null,
  "server_artifacts": []
}
```

终态 completed：

```json
{
  "ready": true,
  "status": "completed",
  "result_summary": "...",
  "content": "...",
  "server_artifacts": []
}
```

failed / timeout：

```json
{
  "ready": true,
  "status": "timeout",
  "isError": true,
  "error": {
    "code": "TASK_TIMEOUT",
    "message": "任务执行超时"
  }
}
```

### F7. Artifact API Pull-only 保持一致

当前 `/tasks/{task_id}/artifacts` 会返回 `server_artifacts` 和 `artifact_mode=pull_only`。

v6.3 要求：

```text
1. artifact_mode 永远返回 pull_only。
2. completed 前返回 server_artifacts=[]。
3. completed 后优先返回 task.server_artifacts。
4. 如果 task.server_artifacts 为空，则 fallback 到 materialized artifact。
5. Desktop 只预览 / 下载 / 导入，不直接读取远端 workspace。
```

### F8. 任务去重与重复调用保护

MCP Gateway 当前会注入 request fingerprint，用于任务去重。

v6.3 要求：

```text
1. Desktop 同一 conversation_id + tool_name + prompt + context 生成稳定 fingerprint。
2. 10 秒内重复提交相同任务时返回 existing task。
3. async_event dedupe 命中时返回原 task 的 event_stream / result_url / artifact_url。
4. 避免用户重复点击导致两个相同任务同时 running。
```

### F9. Expert Invocation Log 与 HermesTask 对齐

当前 Worker 在 `expert_agent_event_stream` 场景下会同步 expert invocation log。

v6.3 要求 Runtime Skill 固定路由也写入调用日志，字段统一：

```text
expert_slug / tool_name
task_id / task_no
execution_mode = async_event
orchestration_mode = runtime_skill_fixed_route
stream_mode = nodeskclaw_task_sse
status
duration_ms
error_code
```

这样 Desktop 的“专家调用日志”页面不会只看到旧 Expert Gateway 任务，也能看到 Runtime Skill 任务。

## 10. Worker 修复要求

### W1. Runtime Skill 只走 `_execute_api_server_task`

`route_type=hermes_api_server` 必须进入 `_execute_api_server_task()`。该路径会从 arguments 中读取 `prompt` 与 `context`，并调用 `execute_runtime_skill_via_api_server()`。

### W2. `/v1/chat/completions` 结果写入完整事件

`execute_runtime_skill_via_api_server()` 当前通过 `/v1/chat/completions` 调用 Hermes Runtime Skill。

v6.3 要求 Worker 在以下节点写事件：

```text
before call:
  hermes.run.started

after success:
  hermes.run.completed
  task.completed

after failure:
  hermes.run.failed
  task.failed

artifact materialized:
  artifact.ready
```

完成后继续执行 server artifact materialization。当前源码已支持发现产物、fallback materialize、写入 server_artifacts、发布 artifact_ready。

### W3. Timeout 策略明确化

```text
1. route_snapshot.timeout_seconds 优先。
2. installation 默认 timeout 次之。
3. 系统默认 HERMES_TASK_TIMEOUT_SECONDS 兜底。
4. 超时必须写 task.timeout 事件。
5. 超时后 result API 返回 ready=true + isError=true。
```

## 11. 前后端协作约定

Desktop v7.5 调用顺序：

```text
1. tools/list
2. tools/call
3. 从 structuredContent 读取 task_id / event_stream / result_url / artifact_url
4. EventSource(event_stream)
5. running 阶段可调用 snapshot / timeline
6. task.completed 后调用 result_url
7. 调用 artifact_url 展示中心产物
```

Desktop 不再：

```text
1. 不调用 /expert-gateway 旧链路发起 Runtime Skill。
2. 不向 arguments 写 _routing / _execution / route_config。
3. 不直接调用 Hermes Agent /v1/runs。
4. 不等 result API 完成后才打开任务明细。
```

## 12. 验收标准

### A1. 相同 Skill 调用一致性


测试请求包含 `_routing`：

```json

  }
```

```text

2. 不创建 HermesTask。
3. MCP audit 记录 failed。
```

### A3. SSE Timeline 可用

```text
1. tools/call 返回 event_stream。
2. Desktop 立即连接 event_stream。
3. 运行中每 15-30 秒有 heartbeat。
4. 至少出现 task.created / task.accepted / hermes.run.started。
```

### A4. 任务明细运行中可打开

```text
1. queued 状态打开明细不报错。
```

```text
1. completed 前 artifacts 返回空数组。
2. completed 后返回 server_artifacts。
3. artifact_mode=pull_only。
```
