# NoDeskClaw Expert MCP Gateway 方案 PRD

版本：v6.0
主题：NoDeskClaw Expert MCP Gateway / Copilot Desktop Remote Experts
适用范围：nodeskclaw-backend、nodeskclaw portal `/hermes/agents`、copilot-desktop `screen/Hermes`、Docker Hermes Agent Runtime
状态：方案确认版
核心变更：不使用 HermesTask；新增 `/api/v1/expert/*` 面向 Desktop 的专家网关；通过轻量调用日志支撑后台专家日志查看。

---

# 1. 背景

当前 nodeskclaw 已具备 Hermes Agent Docker 实例管理能力，并且 `/hermes/agents` 页面已经可以看到实例状态、API Server 状态、MCP Env 写入状态、Router 同步状态、测试调用能力。

同时，现有实例级 MCP 接口已经可用：

```text
POST /api/v1/hermes/mcp/{agent_alias}
```

例如：

```text
POST /api/v1/hermes/mcp/common-writer
```

支持：

```json
{
  "jsonrpc": "2.0",
  "id": "tools-list-1",
  "method": "tools/list"
}
```

以及：

```json
{
  "jsonrpc": "2.0",
  "id": "call-1",
  "method": "tools/call",
  "params": {
    "name": "some_skill_tool",
    "arguments": {
      "prompt": "..."
    }
  }
}
```

v6.0 的目标是：在现有 Hermes MCP 能力之上，增加一层面向 copilot-desktop 的 Expert MCP Gateway。

---

# 2. 核心结论

v6.0 采用以下执行模型：

```text
copilot-desktop
  ↓
/api/v1/expert/health
/api/v1/expert/mcp
/api/v1/expert/mcp/{expert_slug}
  ↓
nodeskclaw Expert MCP Gateway
  ↓
expert / expert_skill / expert_team 配置映射
  ↓
/api/v1/hermes/mcp/{agent_alias}
  ↓
Docker Hermes Agent API_SERVER
```

v6.0 不使用 HermesTask。

调用结果直接由 `/api/v1/expert/mcp/{expert_slug}` 返回给 desktop。

后台只增加轻量调用日志：

```text
expert_invocation_log
```

用于：

```text
1. 专家调用日志查看
2. 专家调用明细查看
3. 错误诊断
4. 耗时统计
5. 用户审计
```

---

# 3. 目标

## 3.1 产品目标

在 nodeskclaw 后台实现：

```text
1. 在 /hermes/agents 页面将 AI 员工设置为专家。
2. 设置为专家后，写入 expert 表。
3. expert 表只引用 hermes_agent_id，不复制 Hermes 实例完整配置。
4. 专家明细中展示该 Hermes 实例下的 skill/tool 清单。
5. 每个 skill 可以设置 public 和 call_enabled。
6. public=true 后，Desktop 能看到该专家能力。
7. call_enabled=true 后，Desktop 能调用该专家能力。
8. 每次调用写入轻量调用日志。
9. 后台可以查看专家调用日志和调用明细。
```

在 copilot-desktop 中实现：

```text
1. 通过 /api/v1/expert/health 获取 Expert Gateway 健康状态。
2. 通过 /api/v1/expert/mcp 获取专家清单。
3. 通过 /api/v1/expert/mcp/{expert_slug} 获取专家能力清单。
4. 通过 /api/v1/expert/mcp/{expert_slug} 调用专家能力。
5. Desktop 不直接调用 /api/v1/hermes/mcp/{agent_alias}。
```

## 3.2 技术目标

```text
1. 复用现有 /api/v1/hermes/mcp/{agent_alias}。
2. 新增 /api/v1/expert/* 作为 Desktop 专用稳定接口。
3. expert_slug 是专家产品标识，不等同于 agent_alias。
4. Expert Gateway 负责 expert_slug 到 agent_alias 的映射。
5. Expert Gateway 负责 skill_name 到 upstream_tool_name 的映射。
6. Expert Gateway 负责权限、公开状态、调用状态、日志、错误封装。
7. 禁止 Desktop 传入 route override 参数。
8. 不引入 HermesTask、队列、worker、timeline 机制。
```

# 4. 不做范围

v6.0 不做：

```text
1. 不引入 HermesTask。
2. 不实现复杂任务队列。
3. 不实现任务 timeline。
4. 不实现中心 Artifact 管理。
5. 不让 Desktop 直接选择 Runtime。
6. 不让 Desktop 直接调用 /api/v1/hermes/mcp/{agent_alias}。
7. 不让 Desktop 传 _routing / _execution / route_config。
8. 不复制 Hermes Agent 实例完整配置到 expert 表。
9. 不在 Desktop 本地安装专家 Profile。
```

# 5. 术语定义

## 5.1 Hermes Agent

Docker 中实际运行的 Hermes Agent 实例。

示例：

```text
common-writer
common-researcher
common-writer-zh
```

由 `/hermes/agents` 管理。

## 5.2 Agent Alias

Hermes 实例在 nodeskclaw 内部的运行别名。

示例：

```text
common-writer
```

用于内部调用：

```text
/api/v1/hermes/mcp/common-writer
```

## 5.3 Expert

面向 Desktop 发布的专家产品对象。

Expert 引用某个 Hermes Agent，但不是 Hermes Agent 的完整复制。

```text
expert.hermes_agent_id -> hermes_agents.id
```

## 5.4 Expert Slug

面向 Desktop 的专家标识。

示例：

```text
writing-expert
market-research-expert
common-writer
```

第一阶段可以让：

```text
expert_slug = common-writer
agent_alias = common-writer
```

但代码层必须支持两者不相等。

## 5.5 Expert Skill

专家公开给 Desktop 的能力。

Expert Skill 映射到底层 Hermes MCP tool。

```text
expert_skill.skill_name -> Desktop 可见能力名
expert_skill.upstream_tool_name -> /api/v1/hermes/mcp/{agent_alias} 的真实 tool name
```

## 5.6 Public

控制能力是否在 Desktop 可见。

```text
public = true
  Desktop tools/list 可见

public = false
  Desktop tools/list 不可见
```

## 5.7 Call Enabled

控制能力是否允许被 Desktop 调用。

```text
call_enabled = true
  Desktop 可以 tools/call

call_enabled = false
  Desktop 可见但不能调用，或直接不可见，取决于 public
```

## 5.8 Expert Invocation Log

专家调用日志。

用于后台审计和诊断，不承担任务队列职责。

# 7. 后端数据模型

## 7.1 `hermes_agents`

现有表继续作为 Hermes Agent 实例真相源。

建议保留字段职责：

```text
id
org_id
agent_alias
container_name
webui_url
api_server_url
gateway_url
model_name
docker_status
api_server_status
agent_callable
runtime_ready
mcp_env_status
router_sync_status
last_seen_at
created_at
updated_at
```

v6.0 不要求在 `hermes_agents` 中塞入过多专家字段。专家产品信息进入 `expert` 表。

可选增加：

```sql
ALTER TABLE hermes_agents ADD COLUMN expert_enabled BOOLEAN DEFAULT false;
```

该字段只用于后台卡片快速显示，不作为 Desktop 读取专家目录的唯一判断依据。

---

## 7.2 `expert`

专家目录表。

```sql
CREATE TABLE expert (
  id UUID PRIMARY KEY,
  org_id UUID NOT NULL,

  hermes_agent_id UUID NOT NULL,

  expert_slug VARCHAR(128) NOT NULL,
  display_name VARCHAR(128) NOT NULL,
  description TEXT,

  category VARCHAR(64),
  tags JSONB DEFAULT '[]',
  avatar VARCHAR(128),

  published BOOLEAN DEFAULT false,
  enabled BOOLEAN DEFAULT true,

  sort_order INT DEFAULT 100,

  created_by UUID,
  updated_by UUID,
  created_at TIMESTAMP NOT NULL,
  updated_at TIMESTAMP NOT NULL,

  UNIQUE(org_id, expert_slug)
);
```

字段说明：

```text
hermes_agent_id
  引用 hermes_agents.id。

expert_slug
  Desktop 调用路径中的专家标识。

display_name
  Desktop 展示名称。

published
  是否发布到 Desktop。

enabled
  是否启用专家。

sort_order
  Desktop 展示排序。
```

示例：

```text
expert_slug: common-writer
display_name: 生文专家
hermes_agent_id: <common-writer-agent-id>
published: true
enabled: true
```

---

## 7.3 `expert_skill`

专家能力表。

```sql
CREATE TABLE expert_skill (
  id UUID PRIMARY KEY,
  org_id UUID NOT NULL,
  expert_id UUID NOT NULL,

  skill_name VARCHAR(128) NOT NULL,
  upstream_tool_name VARCHAR(255) NOT NULL,

  display_name VARCHAR(128),
  description TEXT,
  input_schema JSONB DEFAULT '{}',

  public BOOLEAN DEFAULT false,
  call_enabled BOOLEAN DEFAULT false,

  risk_level VARCHAR(32) DEFAULT 'low',
  approval_mode VARCHAR(32) DEFAULT 'server',
  output_formats JSONB DEFAULT '["markdown"]',

  sort_order INT DEFAULT 100,

  created_by UUID,
  updated_by UUID,
  created_at TIMESTAMP NOT NULL,
  updated_at TIMESTAMP NOT NULL,

  UNIQUE(org_id, expert_id, skill_name),
  UNIQUE(org_id, expert_id, upstream_tool_name)
);
```

字段说明：

```text
skill_name
  Desktop 可见和调用的能力名。

upstream_tool_name
  转发到 /api/v1/hermes/mcp/{agent_alias} 时使用的真实 tool name。

public
  是否在 Desktop tools/list 中可见。

call_enabled
  是否允许 Desktop tools/call。

risk_level
  low / medium / high。

approval_mode
  none / server / admin。
```

状态规则：

```text
public=false, call_enabled=false
  Desktop 不可见，不可调用。

public=true, call_enabled=false
  Desktop 可见，但不可调用。

public=true, call_enabled=true
  Desktop 可见，可调用。

public=false, call_enabled=true
  非法状态。保存时自动纠正为 call_enabled=false。
```

## 7.6 `expert_invocation_log`

轻量调用日志表。

```sql
CREATE TABLE expert_invocation_log (
  id UUID PRIMARY KEY,
  org_id UUID NOT NULL,
  user_id UUID,

  expert_id UUID,
  expert_skill_id UUID,
  expert_team_id UUID,

  expert_slug VARCHAR(128),
  skill_name VARCHAR(128),
  upstream_tool_name VARCHAR(255),
  agent_alias VARCHAR(128),

  request_id VARCHAR(128),
  jsonrpc_id VARCHAR(128),

  status VARCHAR(32) NOT NULL,
  -- started | completed | failed | rejected | timeout

  request_payload JSONB,
  request_prompt_preview TEXT,

  response_preview TEXT,
  response_content_type VARCHAR(64),
  response_size_bytes INT,

  error_code VARCHAR(128),
  error_message TEXT,
  error_detail JSONB,

  started_at TIMESTAMP NOT NULL,
  finished_at TIMESTAMP,
  duration_ms INT,

  client_source VARCHAR(64),
  client_version VARCHAR(64),
  client_device_id VARCHAR(128),

  created_at TIMESTAMP NOT NULL
);
```

日志原则：

```text
1. 不记录 token。
2. 不记录 Authorization header。
3. request_payload 需要脱敏。
4. response_preview 默认截断，例如 4000 字符。
5. 大结果不直接写入日志全文。
6. 错误 detail 可写入 JSONB，便于后台诊断。
```

# 8. 后台 `/hermes/agents` 产品改造

## 8.1 Agent 卡片新增状态

每个 Hermes Agent 卡片增加：

```text
专家状态：
  未设置为专家
  已设置为专家，未发布
  已发布到 Desktop
  已禁用

专家 slug：
  common-writer

公开能力：
  3 / 8

可调用能力：
  2 / 8

最近调用：
  20 次 / 24h
```

## 8.2 Agent 卡片新增操作

```text
设置为专家
编辑专家
发布到 Desktop
下架专家
查看专家能力
查看调用日志
```

## 8.3 设置为专家流程

```text
用户在 /hermes/agents 点击“设置为专家”
  ↓
打开专家配置 Drawer
  ↓
填写：
    expert_slug
    display_name
    description
    category
    tags
    avatar
  ↓
保存
  ↓
创建 expert 记录：
    expert.hermes_agent_id = hermes_agents.id
  ↓
默认 published=false
  ↓
进入专家能力配置页
```

## 8.4 专家能力配置流程

```text
用户点击“查看专家能力”
  ↓
nodeskclaw 调用：
  POST /api/v1/hermes/mcp/{agent_alias}
  method = tools/list
  ↓
读取该 Hermes 实例真实 tools
  ↓
同步到 expert_skill 候选列表
  ↓
用户逐个设置：
    public
    call_enabled
    display_name
    description
    risk_level
    approval_mode
  ↓
保存 expert_skill
```

## 8.5 能力状态规则

后台能力列表显示：

```text
未公开
  public=false
  call_enabled=false

已公开，不可调用
  public=true
  call_enabled=false

已公开，可调用
  public=true
  call_enabled=true
```

## 8.6 发布专家前置条件

专家发布到 Desktop 必须满足：

```text
1. Hermes Agent Docker running。
2. API Server online。
3. Agent callable。
4. Runtime ready。
5. MCP Env 已写入或已授权。
6. Router 已同步。
7. expert_slug 已配置。
8. display_name 已配置。
9. 至少有一个 expert_skill.public=true。
```

如果存在 `public=true` 但 `call_enabled=false`，允许发布，但 Desktop 中该能力显示为不可调用。

# 9. Desktop API 设计

## 9.1 健康检查

```http
GET /api/v1/expert/health
```

返回：

```json
{
  "ok": true,
  "status": "healthy",
  "gateway": {
    "name": "expert-mcp-gateway",
    "version": "v6.0"
  },
  "catalog": {
    "publishedExperts": 3,
    "publicSkills": 12,
    "callableSkills": 8
  },
  "runtimes": [
    {
      "expertSlug": "common-writer",
      "displayName": "生文专家",
      "status": "ready",
      "agentAlias": "common-writer",
      "apiServer": "online",
      "agentCallable": true,
      "runtimeReady": true
    }
  ]
}
```

说明：

```text
1. health 面向 Desktop。
2. 不返回敏感 token。
3. 可以返回 expertSlug。
4. agentAlias 可选返回；如果返回，仅作为诊断字段，不作为 Desktop 调用依据。
5. health 不应每次强制实时 tools/list 所有实例。
```

建议缓存：

```text
expert health cache TTL = 30s
runtime readiness cache TTL = 30s
```

### 9.3.1 tools/list：获取专家能力

请求：

```json
{
  "jsonrpc": "2.0",

```

返回：

```json
{
  "jsonrpc": "2.0",
  "id": "skills-list-1",
  "result": {
    "tools": [
      {
        "name": "report-writing",
        "description": "生成结构化报告。",
        "inputSchema": {
          "type": "object",
          "required": ["prompt"],
          "properties": {
            "prompt": {
              "type": "string",
              "title": "任务要求"
            },
            "context": {
              "type": "object"
            }
          }
        },
        "annotations": {
          "kind": "expert_skill",
          "expertSlug": "common-writer",
          "displayName": "报告写作",
          "public": true,
          "callEnabled": true,
          "riskLevel": "low",
          "approvalMode": "server",
          "outputFormats": ["markdown"]
        }
      }
    ]
  }
```

过滤规则：

```text

  expert.enabled=true
```

```json
"callEnabled": false
```

Desktop 对该能力显示为“不可调用”。

### 9.3.2 tools/call：调用专家能力

请求：

```json
{
  "jsonrpc": "2.0",
  "id": "call-1",
  "method": "tools/call",
  "params": {
    "name": "report-writing",
    "arguments": {
      "prompt": "请生成一份研华科技客户画像分析报告。",
      "context": {
        "source": "copilot-desktop",
        "conversationId": "local-session-001",
        "workspaceId": "default"
      }
    }
  }
}
```

处理流程：

```text
1. 根据 expert_slug 查询 expert。
2. 校验 expert.published=true。
3. 校验 expert.enabled=true。
4. 根据 params.name 查询 expert_skill.skill_name。
5. 校验 expert_skill.public=true。
6. 校验 expert_skill.call_enabled=true。
7. 校验用户有 expert:invoke 权限。
8. 校验 arguments 不包含 route override 字段。
9. 获取 expert.hermes_agent_id。
10. 获取 hermes_agents.agent_alias。
11. 将 skill_name 映射为 upstream_tool_name。
12. 写入 expert_invocation_log status=started。
13. 内部调用 /api/v1/hermes/mcp/{agent_alias} tools/call。
14. 成功后更新 expert_invocation_log status=completed。
15. 失败后更新 expert_invocation_log status=failed。
16. 返回 MCP result。
```

内部转发请求：

```json
{
  "jsonrpc": "2.0",
  "id": "call-1",

  "params": {
```

# 22. 最终确认

v6.0 最终方案为：

```text
1. /api/v1/expert/* 是 Desktop 专用 Expert Gateway。
2. /api/v1/hermes/mcp/{agent_alias} 是 nodeskclaw 内部 Hermes Runtime MCP 入口。
3. Desktop 只看到 expert_slug，不直接控制 agent_alias。
4. Expert 表只引用 hermes_agent_id。
5. Expert Skill 使用 public 控制可见性，call_enabled 控制可调用性。
6. 不使用 HermesTask。
7. 使用 expert_invocation_log 支撑后台专家日志和诊断。
8. 后续专家团队由 Expert Gateway 编排，不由 Desktop 编排。
```
