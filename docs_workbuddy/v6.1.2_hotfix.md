# NoDeskClaw Expert MCP Skill 单开关修复方案 PRD

版本：v6.1.2_hotfix
主题：Expert MCP 能力配置单开关逻辑修复
适用范围：nodeskclaw-backend、nodeskclaw-portal `/hermes/agents`、Expert MCP Gateway
状态：Cursor 实施版
基于版本：v6.1.1_hotfix
核心结论：Portal 能力配置页采用“一个开关 = 公开并可调用”的 MVP 模式；开启 skill 时同时设置 `public=true` 与 `call_enabled=true`，关闭 skill 时同时设置 `public=false` 与 `call_enabled=false`。

---

# 1. 背景

当前 `/hermes/agents` 页面中，管理员进入某个专家的“查看能力”抽屉后，可以看到上游 Hermes Agent 同步出来的 tools，例如：

```text
hermes_xieyi__customer-profiling
hermes_xieyi__xurl
hermes_xieyi__frontend-design
```

但是点击右侧 Switch 后，没有触发后台接口请求，导致：

```text
公开 0/108
可调用 0/108
```

无法变化，最终发布专家时报错：

```text
no_public_skill
```

当前产品阶段不需要把“公开”和“可调用”拆成两个独立开关。为了快速打通 Expert MCP 发布和 Desktop 调用链路，采用 MVP 方案：

```text
一个开关 = 公开并可调用
```

---

# 2. 修复目标

## 2.1 产品目标

在 Expert 能力配置页面，管理员只需要一个开关即可控制该 skill 是否对 Desktop 可见并可调用。

开启时：

```json
{
  "public": true,
  "call_enabled": true
}
```

关闭时：

```json
{
  "public": false,
  "call_enabled": false
}
```

前端展示逻辑：

```text
Switch ON
  表示该能力已公开到 Desktop，且允许调用。

Switch OFF
  表示该能力不在 Desktop 展示，也不可调用。
```

---

## 2.2 技术目标

修复以下问题：

```text
1. Portal 点击 skill 开关必须触发 PATCH 请求。
2. 后端必须保存 public 与 call_enabled。
3. 开启开关时 public=true 且 call_enabled=true。
4. 关闭开关时 public=false 且 call_enabled=false。
5. 请求成功后刷新能力列表与 Agent 卡片统计。
6. 请求失败后恢复开关状态并提示错误。
7. Expert 发布只要求至少一个 public=true 的 skill。
8. Expert MCP 发布不依赖旧 MCP Skill Gateway 授权和 Router 同步。
```

---

# 3. 不做范围

本次修复不做：

```text
1. 不拆成“公开”和“可调用”两个开关。
2. 不改 Desktop 调用接口。
3. 不改 /api/v1/expert/mcp/{slug} 协议。
4. 不引入 HermesTask。
5. 不要求旧 MCP Skill Gateway 授权。
6. 不要求旧 MCP Skill Router 同步。
7. 不删除 public / call_enabled 字段。
8. 不删除后端对 public/call_enabled 的独立校验能力。
```

说明：

```text
后端仍然保留 public 和 call_enabled 两个字段。
只是 Portal 当前版本用一个 Switch 同时控制两个字段。
```

# 4. 状态定义

## 4.1 字段含义

```text
public
  控制 Desktop 是否能在 tools/list 中看到该 skill。

call_enabled
  控制 Desktop 是否能对该 skill 执行 tools/call。
```

## 4.2 MVP 单开关状态映射

| Switch | public | call_enabled | Desktop 可见 | Desktop 可调用 |
| ------ | -----: | -----------: | ---------: | ----------: |
| OFF    |  false |        false |          否 |           否 |
| ON     |   true |         true |          是 |           是 |

## 4.3 禁止状态

MVP 下 Portal 不产生以下状态：

```text
public=true, call_enabled=false
```

该状态后端可以保留支持，但 Portal 当前不提供入口。

也禁止出现：

```text
public=false, call_enabled=true
```

后端必须自动纠正。

# 6. 后端归一化规则

后端必须保底归一化，不能只依赖前端。

## 6.1 规则

```text
public=false
  => call_enabled=false

call_enabled=true
  => public=true
```

## 6.2 推荐实现

```python
def normalize_skill_visibility_update(current, patch):
    next_public = current.public
    next_call_enabled = current.call_enabled

    if "public" in patch:
        next_public = patch["public"]

    if "call_enabled" in patch:
        next_call_enabled = patch["call_enabled"]

    # 关闭公开时，必须关闭调用
    if next_public is False:
        next_call_enabled = False

    # 开启调用时，必须自动公开
    if next_call_enabled is True:
        next_public = True

    return next_public, next_call_enabled
```

## 6.3 单开关接口推荐简化

如果 Portal 单开关只传 `enabled`，后端也可以提供专用接口：

```http
PATCH /api/v1/expert/expert-skills/{skill_id}/visibility
```

请求：

```json
{
  "enabled": true
}
```

后端转换为：

```json
{
  "public": true,
  "call_enabled": true
}
```

关闭：

```json
{
  "enabled": false
}
```

后端转换为：

```json
{
  "public": false,
  "call_enabled": false
}
```

但为了少改接口，优先使用现有 PATCH 接口并传：

```json
{
  "public": true,
  "call_enabled": true
}
```

# 7. Portal 前端修复

## 7.1 当前问题

当前 Switch 点击没有任何 Network 请求，可能原因：

```text
1. Switch 只用了 defaultChecked。
2. 没有绑定 @update:checked。
3. 绑定了本地状态，但没有调用 API。
4. expertCatalog.ts 没有 updateExpertSkill 方法。
5. 前端 API 路径与后端实际路径不一致。
```

---

## 7.2 正确 UI 行为

在 `ExpertSkillsSheet.vue` 中，每个 skill 只有一个开关。

开关状态：

```ts
const checked = skill.public === true && skill.call_enabled === true;
```

开启：

```ts
{
  public: true,
  call_enabled: true
}
```

关闭：

```ts
{
  public: false,
  call_enabled: false
}
```

---

## 7.3 Vue 组件修复示例

```vue
<Switch
  :checked="skill.public && skill.call_enabled"
  :disabled="skill._updating"
  @update:checked="checked => handleToggleSkill(skill, checked)"
/>
```

不要只使用：

```vue
<Switch :default-checked="skill.public && skill.call_enabled" />
```

`defaultChecked` 只负责初始化，不会提交状态。

---

## 7.4 前端处理函数

```ts
async function handleToggleSkill(skill: ExpertSkillRow, checked: boolean) {
  const previousPublic = skill.public;
  const previousCallEnabled = skill.call_enabled;

  try {
    skill._updating = true;

    // 乐观更新
    skill.public = checked;
    skill.call_enabled = checked;

    await updateExpertSkill(skill.id, {
      public: checked,
      call_enabled: checked
    });

    toast.success(
      checked
        ? "能力已公开并启用调用"
        : "能力已关闭"
    );

    await reloadSkills();
    await refreshAgentCard();
  } catch (error) {
    // 失败回滚
    skill.public = previousPublic;
    skill.call_enabled = previousCallEnabled;

    toast.error(resolveApiErrorMessage(error));
  } finally {
    skill._updating = false;
  }
}
```

# 10. Expert 发布逻辑配合

## 10.1 发布前置条件

Expert 发布到 Desktop 只需要：

```text
1. Docker running。
2. API Server online。
3. Agent callable。
4. Runtime ready。
5. expert_slug 非空。
6. display_name 非空。
7. 至少一个 expert_skill.public=true。
```

由于单开关开启时会同时设置：

```text
public=true
call_enabled=true
```

所以开启任意一个 skill 后，发布条件即可满足。

## 10.2 不再校验

发布专家时不得校验：

```text
1. mcp_env_not_synced。
2. router_not_synced。
3. 旧 MCP Skill Gateway 授权。
4. 旧 MCP Skill Router 同步。
```
