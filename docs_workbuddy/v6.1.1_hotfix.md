# NoDeskClaw Expert MCP 发布与 Skill 开关逻辑修复 PRD

版本：v6.1.1_hotfix
主题：Expert MCP 发布前置条件与专家能力开启逻辑修复
适用范围：nodeskclaw-backend、nodeskclaw-portal `/hermes/agents`、Expert MCP Gateway
状态：Cursor 实施版
基于版本：Expert MCP Gateway v6.1
核心结论：Expert MCP 发布不应依赖旧 MCP Skill Gateway 授权状态；发布专家只依赖 Hermes Agent Runtime 可调用、专家基础信息完整、至少一个 public skill；skill 开启逻辑以 `public` 和 `call_enabled` 两个字段为准。

---

# 1. 背景

当前在 `/hermes/agents` 页面将 Docker Hermes Agent 实例设置为专家后，点击“发布到 Desktop”时出现错误：

```text
专家发布前置条件未满足:
mcp_env_not_synced,
router_not_synced,
no_public_skill
```

其中：

```text
mcp_env_not_synced
router_not_synced
```

属于旧链路状态：

```text
Docker 内 hermes-agent
  ↓
调用 nodeskclaw 旧 MCP Skill Gateway
```

该链路表示：容器内 Hermes Agent 是否被授权调用 nodeskclaw 旧的组织级 MCP Skill Gateway。

但 v6.1 Expert MCP Gateway 的链路是：

```text
copilot-desktop
  ↓
/api/v1/expert/mcp/{slug}
  ↓
nodeskclaw Expert MCP Gateway
  ↓
dispatch_agent_mcp(profile_name)
  ↓
Docker Hermes Agent API_SERVER
```

因此，Expert MCP 发布到 Desktop 不应该依赖旧 MCP Skill Gateway 授权状态。

当前真正应该阻止发布的条件是：

```text
no_public_skill
```

即：该专家还没有任何公开能力给 Desktop。

---

# 2. 修复目标

## 2.1 产品目标

修复后，管理员在 `/hermes/agents` 页面完成以下操作即可发布专家：

```text
1. 设置 Hermes Agent 实例为专家。
2. 打开“查看能力”。
3. 同步或确认该实例下的 tools。
4. 将至少一个 skill 设置为 public=true。
5. 可选将该 skill 设置为 call_enabled=true。
6. 点击“发布到 Desktop”。
```

不再要求：

```text
1. 授权 MCP Skill Gateway。
2. 同步 MCP Skill Router。
3. 容器内写入 NODESKCLAW_MCP_TOKEN。
```

## 2.2 技术目标

修复以下逻辑：

```text
1. Expert 发布校验移除 mcp_env_not_synced。
2. Expert 发布校验移除 router_not_synced。
3. ExpertTeam upstream_skill 模式发布也不校验旧 MCP Gateway 授权和 Router 同步。
4. Expert 发布只校验 Expert MCP 所需条件。
5. Skill 开启逻辑明确 public 与 call_enabled 的状态关系。
6. Portal UI 将“旧 MCP Skill Gateway 状态”和“Expert MCP 发布状态”分区展示。
7. 发布失败提示只显示 Expert MCP 相关原因。
```

# 3. 概念边界

## 3.1 旧 MCP Skill Gateway

用途：

```text
让 Docker 容器内的 hermes-agent 作为 MCP Client，调用 nodeskclaw 旧组织级 MCP Skill Gateway。
```

相关状态：

```text
MCP：未授权 / Env 已写入 / 已授权
Router：MCP 未授权 / 已同步
NODESKCLAW_MCP_URL
NODESKCLAW_MCP_TOKEN
NODESKCLAW_MCP_ENABLED
```

这些状态不属于 Expert MCP 发布前置条件。

## 3.2 Expert MCP Gateway

用途：

```text
让 copilot-desktop 通过 nodeskclaw 调用远端 Hermes Agent 的 skill。
```

调用链路：

```text
copilot-desktop
  ↓
/api/v1/expert/mcp/{slug}
  ↓
Expert Gateway
  ↓
dispatch_agent_mcp(profile_name)
  ↓
Hermes Agent API_SERVER
```

Expert MCP 发布只需要确认：

```text
1. 绑定的 Hermes Agent Runtime 可用。
2. 专家本身信息完整。
3. 至少有一个 public skill。
```

# 6. Expert 发布前置条件修复

## 6.1 当前错误逻辑

当前发布校验错误地混入了旧 MCP Skill Gateway 状态：

```text
mcp_env_not_synced
router_not_synced
```

这会导致以下情况无法发布：

```text
Docker running
API Server online
Agent callable
Runtime ready
tools/list 能同步到 108 个 tools
但旧 MCP Gateway 未授权
```

这是错误的。

## 6.2 新的 Expert 发布前置条件

发布 Expert 到 Desktop 只校验：

```text
1. Docker running。
2. API Server online。
3. Agent callable。
4. Runtime ready。
5. expert_slug 非空。
6. display_name 非空。
7. 至少 1 个 expert_skill.public = true。
```

不再校验：

```text
1. mcp_env_synced。
2. mcp_env_status。
3. mcp_authorized。
4. router_synced。
5. router_status。
6. NODESKCLAW_MCP_TOKEN。
7. NODESKCLAW_MCP_URL。
```

## 6.3 新的 ExpertTeam 发布前置条件

### upstream_skill 模式

当：

```text
expert_team.orchestration_mode = upstream_skill
```

发布只校验：

```text
1. 绑定的 Hermes Agent Docker running。
2. API Server online。
3. Agent callable。
4. Runtime ready。
5. team_slug 非空。
6. display_name 非空。
7. 至少 1 个 expert_team_skill.public = true。
```

不校验：

```text
1. 旧 MCP Gateway 授权。
2. 旧 Router 同步。
3. expert_team_members。
4. expert_team_orchestrator 配置。
```

### gateway_sequential 模式

当：

```text
expert_team.orchestration_mode = gateway_sequential
```

发布校验：

```text
1. team_slug 非空。
2. display_name 非空。
3. 至少 2 个 team members。
4. 每个成员 expert enabled=true。
5. 每个成员 expert published=true。
6. 每个成员至少有一个可调用 skill。
```

仍然不校验旧 MCP Gateway 授权和旧 Router 同步。

# 7. 后端发布校验设计

## 7.1 Expert 发布校验伪代码

```python
def validate_expert_publish_preconditions(db, expert):
    errors = []

    agent = expert.hermes_agent

    if not agent:
        errors.append("agent_not_bound")
        return errors

    if agent.docker_status != "running":
        errors.append("docker_not_running")

    if agent.api_server_status != "online":
        errors.append("api_server_not_online")

    if not agent.agent_callable:
        errors.append("agent_not_callable")

    if not agent.runtime_ready:
        errors.append("runtime_not_ready")

    if not expert.expert_slug:
        errors.append("missing_slug")

    if not expert.display_name:
        errors.append("missing_display_name")

    public_skill_count = count_public_expert_skills(db, expert.id)
    if public_skill_count <= 0:
        errors.append("no_public_skill")

    return errors
```

必须删除：

```python
if not agent.mcp_env_synced:
    errors.append("mcp_env_not_synced")

if not agent.router_synced:
    errors.append("router_not_synced")
```

## 7.2 ExpertTeam 发布校验伪代码

```python
def validate_expert_team_publish_preconditions(db, team):
    errors = []

    if not team.team_slug:
        errors.append("missing_slug")

    if not team.display_name:
        errors.append("missing_display_name")

    mode = team.orchestration_mode or "upstream_skill"

    if mode == "upstream_skill":
        agent = team.hermes_agent

        if not agent:
            errors.append("agent_not_bound")
            return errors

        if agent.docker_status != "running":
            errors.append("docker_not_running")

        if agent.api_server_status != "online":
            errors.append("api_server_not_online")

        if not agent.agent_callable:
            errors.append("agent_not_callable")

        if not agent.runtime_ready:
            errors.append("runtime_not_ready")

        public_skill_count = count_public_expert_team_skills(db, team.id)
        if public_skill_count <= 0:
            errors.append("no_public_skill")

    elif mode == "gateway_sequential":
        member_count = count_team_members(db, team.id)
        if member_count < 2:
            errors.append("expert_team_members_required")

        if not all_members_callable(db, team.id):
            errors.append("expert_team_member_skill_not_callable")

    else:
        errors.append("unsupported_orchestration_mode")

    return errors
```

# 8. Skill 开关逻辑修复

## 8.1 字段定义

```text
public:
  是否在 Desktop 的 tools/list 中可见。

call_enabled:
  是否允许 Desktop 对该 skill 执行 tools/call。
```

## 8.2 状态矩阵

| public | call_enabled | Desktop 可见 | Desktop 可调用 | 是否允许      |
| ------ | ------------ | ---------- | ----------- | --------- |
| false  | false        | 否          | 否           | 允许        |
| true   | false        | 是          | 否           | 允许        |
| true   | true         | 是          | 是           | 允许        |
| false  | true         | 不应存在       | 不应存在        | 禁止 / 自动纠正 |

## 8.3 后端强制规则

必须保证：

```text
call_enabled=true => public=true
public=false => call_enabled=false
```

## 8.4 PATCH 更新规则

### 场景 A：只设置 public=false

请求：

```json
{
  "public": false
}
```

后端保存为：

```json
{
  "public": false,
  "call_enabled": false
}
```

### 场景 B：设置 call_enabled=true，但当前 public=false

请求：

```json
{
  "call_enabled": true
}
```

建议后端自动纠正为：

```json
{
  "public": true,
  "call_enabled": true
}
```

原因：

```text
“启用调用”对管理员来说隐含“公开能力”。
```

### 场景 C：同时设置 public=false, call_enabled=true

请求：

```json
{
  "public": false,
  "call_enabled": true
}
```

后端应优先 public=false，保存为：

```json
{
  "public": false,
  "call_enabled": false
}
```

并可返回 warning：

```json
{
  "warning": "call_enabled was disabled because public=false"
}
```

### 场景 D：设置 public=true, call_enabled=false

允许。

```json
{
  "public": true,
  "call_enabled": false
}
```

含义：

```text
Desktop 能看到能力，但按钮不可调用。
```

## 8.5 Skill 开启 API 行为

如果当前实际接口为：

```text
PATCH /api/v1/expert/expert-skills/{skill_id}
```

或：

```text
PATCH /api/v1/expert/experts/{expert_id}/skills/{skill_id}
```

均应执行同一套 normalization：

```python
def normalize_skill_visibility_update(current, patch):
    public = patch.public if "public" in patch else current.public
    call_enabled = patch.call_enabled if "call_enabled" in patch else current.call_enabled

    if public is False:
        call_enabled = False

    if call_enabled is True and public is False:
        public = True

    return public, call_enabled
```

更严格实现：

```python
if patch.get("public") is False:
    public = False
    call_enabled = False
elif patch.get("call_enabled") is True:
    public = True
    call_enabled = True
else:
    public = patch.get("public", current.public)
    call_enabled = patch.get("call_enabled", current.call_enabled)
```

# 9. sync-tools 逻辑修复

## 9.1 sync-tools 不依赖旧 MCP Gateway

同步专家能力时，应调用：

```text
dispatch_agent_mcp(profile_name, body={"method": "tools/list"})
```

它只需要：

```text
1. 绑定 Hermes Agent。
2. API Server online。
3. Agent callable。
4. Runtime ready。
```

不需要：

```text
1. mcp_env_synced。
2. router_synced。
3. 旧 MCP token。
```

## 9.2 sync-tools 前置条件

```text
1. expert.hermes_agent_id 存在。
2. HermesAgentInstance.profile_name 存在。
3. API Server online。
4. Agent callable。
```

如果 Runtime 状态过期，可先触发 refresh status，再执行 sync-tools。

## 9.3 sync-tools 更新规则

同步上游 tools 时：

```text
1. 上游有但本地没有：创建 expert_skill 候选。
2. 上游有且本地已有：更新 input_schema、description、last_synced_at。
3. 不覆盖用户手工设置的 display_name。
4. 不覆盖 public。
5. 不覆盖 call_enabled。
6. 上游已不存在：stale=true。
```

# 10. Expert MCP Catalog 输出规则

## 10.1 /api/v1/expert/mcp tools/list

只有已发布对象进入 Desktop catalog：

```text
expert.published=true
expert.enabled=true
至少一个 expert_skill.public=true
```

不应要求：

```text
mcp_env_synced
router_synced
```

返回示例：

```json
{
  "name": "xieyi-latest",
  "description": "谢艺专家",
  "annotations": {
    "kind": "expert",
    "slug": "xieyi-latest",
    "displayName": "谢艺专家",
    "status": "ready",
    "publicSkillCount": 3,
    "callableSkillCount": 2
  }
}
```

## 10.2 /api/v1/expert/mcp/{slug} tools/list

返回：

```text
public=true 的 skills
```

如果：

```text
public=true
call_enabled=false
```

仍然返回，但：

```json
{
  "annotations": {
    "callEnabled": false
  }
}
```

Desktop 禁用调用按钮。

## 10.3 /api/v1/expert/mcp/{slug} tools/call

调用前必须校验：

```text
1. expert/team published=true。
2. expert/team enabled=true。
3. skill public=true。
4. skill call_enabled=true。
5. 当前用户有 expert_skill:invoke。
6. arguments 无 route override。
```

如果 `call_enabled=false`，返回：

```json
{
  "jsonrpc": "2.0",
  "id": "call-1",
  "error": {
    "code": -32602,
    "message": "Expert skill call is disabled.",
    "data": {
      "errorCode": "EXPERT_SKILL_CALL_DISABLED"
    }
  }
}
```

# 12. Portal UI 调整

## 12.1 状态分区

当前卡片中旧 MCP 状态和 Expert 状态混在一起，容易误导。

建议分区展示：

```text
Expert MCP
  专家：已设置未发布
  公开：0/108
  可调用：0/108
  Desktop：未发布

旧组织 MCP Skill Gateway
  MCP：未授权
  Router：MCP 未授权
```

## 12.2 按钮文案调整

当前按钮：

```text
授权 MCP Skill Gateway
同步 MCP Skill Router
```

建议改为：

```text
授权组织 MCP Skill Gateway
同步组织 MCP Skill Router
```

或：

```text
授权旧 MCP Skill Gateway
同步旧 MCP Skill Router
```

避免和 Expert MCP Gateway 混淆。

## 12.3 发布失败提示调整

当前：

```text
专家发布前置条件未满足: mcp_env_not_synced, router_not_synced, no_public_skill
```

修复后：

```text
专家发布前置条件未满足: no_public_skill
请先进入“查看能力”，至少公开一个能力。
```

如果多个 Expert MCP 条件不满足：

```text
专家发布前置条件未满足:
API Server 未在线、没有公开能力。
```

## 12.4 ExpertSkillsSheet 开关行为

### Public 开关关闭时

```text
1. call_enabled 自动关闭。
2. call_enabled 开关禁用或显示为 false。
```

### Call Enabled 开关打开时

```text
1. 如果 public=false，自动设置 public=true。
2. 同时设置 call_enabled=true。
```

### 推荐 UI 交互

```text
公开开关：
  控制 Desktop 是否可见。

可调用开关：
  控制 Desktop 是否可执行。
  开启可调用时自动公开。
```

# 18. 最终确认

本次修复完成后，Expert MCP 的发布规则为：

```text
Expert MCP 发布不依赖旧 MCP Skill Gateway。
Expert MCP 发布不依赖旧 Router 同步。
Expert MCP 发布只依赖 Runtime 可用、专家信息完整、至少一个 public skill。
```

Skill 开启规则为：

```text
public 控制 Desktop 是否可见。
call_enabled 控制 Desktop 是否可调用。
call_enabled=true 必须 public=true。
public=false 必须 call_enabled=false。
```

最终用户操作路径为：

```text
设置为专家
  ↓
查看能力
  ↓
公开至少一个 skill
  ↓
可选开启调用
  ↓
发布到 Desktop
```
