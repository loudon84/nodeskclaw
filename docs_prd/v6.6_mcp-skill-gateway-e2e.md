# PRD：Copilot Desktop × nodeskclaw MCP Skill Gateway 集成方案

版本：v6.6_mcp-skill-gateway-e2e
范围：copilot-desktop + nodeskclaw
目标：让 Copilot Desktop 通过本地 MCP Proxy 对接 nodeskclaw MCP Skill Gateway，使本地 Hermes Agent 能发现并调用 nodeskclaw 暴露的 Hermes 实例、Skill、GeneHub 工具能力。

---

## 0. 总体定位

### 0.1 当前事实

nodeskclaw 已经完成：

1. 绑定 Docker Hermes Agent 实例。
2. 读取 Hermes 容器运行状态。
3. 读取 WebUI 健康状态。
4. 管理 Hermes skills 列表。
5. 支持安装内置 skill、上传 ZIP、Git 安装。
6. 已有 GeneHub：Gene manifest 中可包含 `skill.name`、`skill.content`、`tool_allow`、`mcp_servers`、`runtime_config`。

copilot-desktop 已经完成：

1. 登录 nodeskclaw backend。
2. 本地 token vault。
3. MCP Skill Gateway Runtime。
4. 本地 Proxy。
5. 将本地 Proxy 注册到 Hermes profile 的 `mcp_servers.mcp_skill_gateway`。
6. 本地 Hermes profile 与 skills 目录管理。
7. Hermes Skills 页面。
8. MCP Gateway 页面基础状态展示。

### 0.2 本阶段目标

本阶段不是做 GeneHub 市场页面，也不是做 skill 发布平台。

本阶段只做一条核心链路：

```text
Copilot Desktop
  → 本地 MCP Proxy
  → nodeskclaw /api/v1/hermes/mcp
  → tools/list
  → tools/call
  → Hermes Agent 可调用 nodeskclaw tools
```

### 0.3 后续阶段关系

```text
v6.6_mcp-skill-gateway-e2e
  先打通 MCP Gateway 调用链路

v6.6_genehub-skill-registration
  再实现 GeneHub skill 注册到 Hermes profile

v6.7_genehub-skill-marketplace
  最后做 Desktop 侧 GeneHub UI、安装、发布、审核
```

# 子 PRD A：copilot-desktop

## A1. 产品目标

在 copilot-desktop 中完成 nodeskclaw MCP Skill Gateway 的端到端接入：

1. Desktop 能从登录配置中识别 nodeskclaw backend。
2. Desktop 启动本地 MCP Proxy。
3. Desktop 本地 Proxy 能转发 JSON-RPC 到 nodeskclaw `/api/v1/hermes/mcp`。
4. Desktop 可执行 MCP `initialize`、`tools/list`、`tools/call` 测试。
5. Desktop 能将本地 Proxy 注册到 Hermes `default` profile。
6. Hermes Gateway 重启后，可通过 Hermes Agent 调用 nodeskclaw MCP tools。
7. Desktop 页面能展示工具数量、工具列表、注册状态、诊断结果与错误原因。

## A6. Hermes Profile 注册规则

注册目标：

```yaml
mcp_servers:
  mcp_skill_gateway:
    type: http
    url: http://127.0.0.1:48742/mcp
    enabled: true
```

禁止：

```text
1. 禁止写入 nodeskclaw backend URL。
2. 禁止写入 accessToken。
3. 禁止 Renderer 直接请求 nodeskclaw MCP。
4. 禁止 Hermes Agent 直接访问远端 nodeskclaw MCP。
```

原因：

```text
所有远端认证只能由 Desktop 本地 Proxy 注入。
Hermes config.yaml 只保存本地 Proxy URL。
```

## A8. 安全要求

1. Token 不进入 Renderer。
2. Token 不进入 Hermes config.yaml。
3. Proxy 只监听 `127.0.0.1`。
4. Proxy request body 限制大小。
5. MCP invoke test 默认只允许只读工具。
6. 写操作工具需要用户确认。
7. 工具调用写入 audit log。
8. 错误日志不输出 Authorization header。

## B5. 权限模型

工具分级：

```text
read:
  hermes.instances.list
  hermes.instance.status
  hermes.skills.list

write:
  hermes.skills.install_builtin
  hermes.skills.install_zip
  hermes.skills.install_git
  hermes.skills.uninstall

admin:
  hermes.instance.restart
  hermes.instance.rebind
```

本阶段只开放 `read`。

权限检查：

```text
1. accessToken 必须有效。
2. 用户必须属于当前 org。
3. 用户必须有访问该 Hermes 实例权限。
4. tools/call 必须检查 tool permission。
5. write/admin tool 必须支持审批或二次确认。
```
