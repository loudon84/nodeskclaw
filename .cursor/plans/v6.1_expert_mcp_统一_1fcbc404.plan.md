# v6.1 Expert MCP Gateway 统一调用规则实施

## 前端表现变化

### 1. Portal 专家团队管理页 - 编排模式切换

**总结**: 专家团队创建/编辑从"纯成员编排"改为"双模式切换"：默认 upstream_skill（绑定 Hermes Agent + 同步能力），可选 gateway_sequential（成员编排）

**元素级变化**:
- 编排模式选择器: **新增**，创建/编辑表单中新增 upstream_skill / gateway_sequential 两个选项，默认选中 upstream_skill
- 绑定 Hermes Agent 下拉: **新增**，upstream_skill 模式下显示，选择团队技能所在的 Agent 实例
- 同步能力按钮: **新增**，upstream_skill 模式下显示，点击后从绑定 Agent 拉取 tools/list
- 团队能力列表: **新增**，upstream_skill 模式下显示已同步的 team skills，可配置 public/call_enabled
- 成员编排区域: **条件显示**，仅 gateway_sequential 模式下显示（原有功能保留）
- 团队列表表格: 新增"编排模式"列和"绑定 Agent"列

**改动前**:
```
+-- 专家团队 -----------------------------------------------+
| [+ 创建团队]                                               |
| Slug | 名称 | 成员数 | 已发布 | 操作                        |
| sale-team | 销售团队 | 3 | 是 | [切换发布] [添加成员]       |
+-----------------------------------------------------------+
-> 只有成员编排模式
```

**改动后**:
```
+-- 专家团队 -----------------------------------------------+
| [+ 创建团队]                                               |
|                                                            |
| +- 创建团队 (展开) ------------------------------------+   |
| | Slug [________]  名称 [________]                     |   |
| | 编排模式:                                             |   |
| |   (o) 上游技能  ( ) 网关顺序编排                      |   |
| | 绑定 Agent: [选择 Agent v]  [同步能力]                |   |
| | +- 团队能力 -----------------------------------+     |   |
| | | skill-name | upstream_tool | public | call   |     |   |
| | +----------------------------------------------+     |   |
| +------------------------------------------------------+   |
|                                                            |
| Slug | 名称 | 编排模式 | Agent | 已发布 | 操作             |
| sale-team | 销售团队 | upstream_skill | agent-1 | 是 | ... |
+-----------------------------------------------------------+
```

### 2. Portal 专家调用日志页 - 新增类型和编排模式筛选

**总结**: 日志筛选栏新增"类型"和"编排模式"两个筛选项，日志列表新增"类型"和"编排模式"两列

**元素级变化**:
- 筛选栏: 新增 catalog_kind 下拉（expert / expert_team / 全部）和 orchestration_mode 下拉（upstream_skill / gateway_sequential / 全部）
- 日志列表: 新增"类型"列（显示 expert 或 expert_team Badge）和"编排模式"列
- 日志详情抽屉: 新增 catalog_kind、catalog_slug、orchestration_mode 字段展示

### Phase 2: 服务层核心 — CatalogResolver + Gateway 重构

#### 2.1 新增 CatalogResolver

新建 `app/services/expert_gateway/catalog_resolver.py`

```python
class CatalogItem:
    id: str
    org_id: str
    kind: Literal["expert", "expert_team"]
    slug: str
    display_name: str
    description: str | None
    hermes_agent_id: str | None
    published: bool
    enabled: bool
    orchestration_mode: str  # expert 固定为 "upstream_skill"
    source_record: Expert | ExpertTeam

class CatalogResolver:
    async def resolve(self, org_id: str, slug: str) -> CatalogItem | None
    # 查询顺序：experts.expert_slug -> expert_teams.team_slug
```

#### 2.2 重构 expert_mcp_gateway_service.py

[nodeskclaw-backend/app/services/expert_gateway/expert_mcp_gateway_service.py](nodeskclaw-backend/app/services/expert_gateway/expert_mcp_gateway_service.py)

核心变更：

- `dispatch_slug` 重命名为 `dispatch_catalog_item`
- 移除现有的 "先查 team 再查 expert" 硬编码逻辑，改用 `CatalogResolver.resolve(slug)`
- `initialize` 响应 version 从 `"v6.0"` -> `"v6.1"`
- `tools/list` 根据 `CatalogItem.kind` 分派：
  - `expert` -> 查 `expert_skills` 表
  - `expert_team` -> 查 `expert_team_skills` 表
- `tools/call` 根据 `kind + orchestration_mode` 分派：
  - `expert` -> 走现有 `_call_expert_skill`（微调参数传递）
  - `expert_team + upstream_skill` -> **新路径**，查 `expert_team_skills`，调用 `ExpertMcpProxyService.call_upstream_tool`
  - `expert_team + gateway_sequential` -> 走现有 `expert_team_orchestrator`
- `_list_catalog_tools` 中 annotations 使用 `slug` 替代 `expertSlug`/`teamSlug`，团队项新增 `orchestrationMode`、`publicSkillCount`、`callableSkillCount`

调用日志记录统一传入 `catalog_kind`、`catalog_slug`、`orchestration_mode`。

#### 2.3 新增 ExpertTeamSkillService

新建 `app/services/expert_gateway/expert_team_skill_service.py`

复用 `ExpertSkillService` 的逻辑模式，关键方法：
- `list_skills(org_id, team_id)` -> `list[ExpertTeamSkillItem]`
- `list_public_skills(org_id, team_id)` -> `list[ExpertTeamSkill]`
- `get_skill_by_name(org_id, team_id, skill_name)` -> `ExpertTeamSkill | None`
- `sync_tools(org_id, user_id, team_id)` -> `ExpertSkillSyncResult`
  - 从 `ExpertTeam.hermes_agent_id` 找到 `profile_name`
  - 调用 `ExpertMcpProxyService.list_upstream_tools`
  - Upsert 到 `expert_team_skills`
- `update_skill(org_id, user_id, skill_id, body)` -> `ExpertTeamSkillItem`
- `build_tool_descriptor(team, skill, runtime_ready)` -> annotations.kind = "expert_team_skill"

## 迁移兼容性

- URL 路径不变（`/api/v1/expert/mcp/{expert_slug}` -> 内部语义变为 `{slug}`）
- 现有 expert 调用不受影响
- 现有 expert_team_orchestrator 代码保留，降级为可选
- 现有 expert_team_members 表保留，gateway_sequential 模式继续使用
- ExpertTeam 表 orchestration_mode 列已存在（v6.0 default = "sequential_gateway"），迁移只需改默认值为 "upstream_skill" 并回填
